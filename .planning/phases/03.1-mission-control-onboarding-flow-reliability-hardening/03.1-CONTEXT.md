# Phase 03.1: Mission Control Onboarding Flow Reliability Hardening - Context

**Gathered:** 2026-03-25
**Status:** Ready for planning
**Source:** Live server investigation of the stalled `Money` board onboarding flow

<domain>
## Phase Boundary

This phase fixes the specific `Mission Control` failure mode where interactive board onboarding appears to hang after a user answer even though the gateway session has already completed.

The phase is limited to `Mission Control` behavior:
- onboarding prompt/dispatch contract
- session selection for onboarding turns
- bounded failure handling in backend/UI
- regression coverage and operator diagnostics

This phase does not redesign the broader `OpenClaw` runtime, security policy, or board-agent governance model.
</domain>

<decisions>
## Implementation Decisions

### Locked Decisions

- Fix the issue on the `Mission Control` side; do not depend on `OpenClaw` changes as a prerequisite.
- Treat the current behavior as a product bug, not as an acceptable edge case.
- `NO_REPLY` must never count as successful progress for interactive `board onboarding`.
- Interactive onboarding must not rely on the same silent session semantics as routine `gateway-main` heartbeat/control-plane turns.
- The user must see an explicit terminal state if onboarding stalls: next question, completion, or actionable error.
- Keep the current sidecar deployment model and dedicated workspace root intact.
- Preserve existing board creation and onboarding data where possible; avoid destructive migrations unless clearly required.

### Claude's Discretion

- Whether to introduce a dedicated session key helper or another isolation mechanism, as long as it breaks the coupling with the routine `agent:mc-gateway-...:main` silent flow.
- Whether bounded failure detection should rely on timeout tracking, session preview/history inspection, or both.
- Exact UI copy for stalled/error onboarding state, provided it is concise and actionable.
</decisions>

<specifics>
## Specific Ideas

- Current observed failing board: `Money`
- Current failing board id: `414b1071-af65-4e35-b33f-f1fdbe43174d`
- Current onboarding session id: `85d57589-cc26-444a-a502-192586f71f43`
- Current gateway session key reused by onboarding: `agent:mc-gateway-b7cc7ef4-c968-4e86-93a2-a63a64335da6:main`
- Observed runtime behavior:
  - first onboarding question posted successfully
  - second onboarding question posted successfully
  - after each subsequent user answer, the gateway session ended with `NO_REPLY`
  - no new `POST /api/v1/agent/boards/{board_id}/onboarding` arrived after the last user answer
  - UI kept polling the onboarding endpoint indefinitely
- Prompt evidence:
  - backend prompt says `Do NOT respond in OpenClaw chat`
  - backend prompt says onboarding responses must go to MC API
  - dispatch uses `deliver=False`
- Root cause hypothesis to preserve:
  - onboarding reuses a silent `gateway-main` session that already treats `NO_REPLY` as a normal non-visible turn completion
</specifics>
