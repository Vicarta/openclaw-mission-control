# Phase 03.1-02 Summary

## Scope

Додати bounded stalled handling, UI recovery path і operator diagnostics, щоб onboarding більше не зависав безкінечно після silent completion або missing callback.

## Implemented

- У `backend/app/models/board_onboarding.py` додано constants для onboarding state machine і callback timeout `20s`.
- У `backend/app/api/board_onboarding.py` додано computed read-state:
  - якщо `status=active`,
  - останнє повідомлення має `role=user`,
  - і `updated_at + 20s < now`,
  - API повертає `status=stalled`.
- У `backend/app/schemas/board_onboarding.py` `BoardOnboardingRead.status` звужено до відомих onboarding states.
- У `frontend/src/components/BoardOnboardingChat.tsx`:
  - polling зупиняється на `stalled`;
  - користувач бачить явний stalled banner;
  - додано recovery action `Retry last step`, який повторно викликає `start` endpoint.
- У `frontend/src/lib/onboarding.ts` додано helper `isOnboardingStalled()`.
- У `frontend/src/components/BoardOnboardingChat.test.tsx` додано regression test на stalled UI/retry path.
- Після верифікації server stack перебілджено й перезапущено:
  - `backend`
  - `webhook-worker`
  - `frontend`

## Outcome

Mission Control onboarding тепер має bounded failure path і recovery UX. Навіть якщо agent callback не приходить, UI більше не крутить endless spinner, а оператор і користувач отримують явний `stalled` стан і наступну дію.
