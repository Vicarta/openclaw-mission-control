# Phase 03.1 Research: Mission Control Onboarding Flow Reliability Hardening

**Date:** 2026-03-25
**Status:** Complete

## Problem Summary

`Mission Control` board creation succeeds, but interactive onboarding can stall indefinitely after a user answer. The backend remains healthy and keeps returning `200` on onboarding polling, while no new onboarding callback arrives from the gateway agent.

## Evidence

### Verified Live Flow

- Board `Money` created successfully at `2026-03-25 14:15:55 UTC`
- Onboarding session `85d57589-cc26-444a-a502-192586f71f43` started successfully
- First question arrived via `POST /api/v1/agent/boards/{board_id}/onboarding`
- First user answer dispatched successfully
- Second question arrived successfully
- Second user answer dispatched successfully
- After the second answer, no new onboarding callback arrived

### Session-Level Finding

The host-side gateway session file showed:

- the session key for onboarding was `agent:mc-gateway-b7cc7ef4-c968-4e86-93a2-a63a64335da6:main`
- this is the same routine gateway-main session family used for heartbeat/control-plane work
- after each successful onboarding callback, the model completed the turn with `NO_REPLY`
- after the final user answer, the model again completed the turn with `NO_REPLY`
- session status was `done`, not `aborted`

This means the system did not hang inside execution. It completed without producing the next onboarding callback.

## Code Findings

### Onboarding Prompt Construction

`backend/app/api/board_onboarding.py` builds a prompt that says:

- `Do NOT respond in OpenClaw chat.`
- `All onboarding responses MUST be sent to Mission Control via API.`

This is correct in principle, but it leaves no safe fallback when the session ends silently.

### Onboarding Dispatch

`backend/app/services/openclaw/onboarding_service.py` dispatches both start and answer messages through:

- `GatewayAgentIdentity.session_key(gateway)`
- `_dispatch_gateway_message(..., deliver=False)`

So interactive onboarding reuses the same `gateway-main` session key and silent delivery mode used for routine control-plane turns.

### Session Identity

`backend/app/services/openclaw/shared.py` defines:

- `session_key(gateway) -> agent:mc-gateway-<gateway_id>:main`

There is no separate onboarding session identity in the current design.

### Bounded Failure Gap

`board_onboarding.answer_onboarding()` dispatches the answer and returns success, but there is no bounded timeout/error transition for the case where the gateway session ends without emitting another onboarding callback.

As a result:

- backend polling continues
- onboarding remains `active`
- UI keeps showing optimistic waiting copy

## Root Cause

The most plausible root cause is architectural coupling:

1. Interactive onboarding is sent into a session designed for silent `gateway-main` control-plane work.
2. That session is invoked with `deliver=False`.
3. The prompt forbids chat replies but does not explicitly forbid silent completion semantics such as `NO_REPLY`.
4. Backend state handling assumes the next step will always be another onboarding callback.
5. When the session silently completes, MC has no bounded failure path and the UI spins forever.

## Recommended Change Directions

1. Introduce a dedicated board-scoped onboarding session contract or equivalent routing isolation.
2. Make `NO_REPLY` invalid for interactive onboarding turns.
3. Add bounded failure detection after `start` and `answer`.
4. Surface stalled/failed state in the UI with retry or inspect instructions.
5. Add regression tests that simulate `NO_REPLY` / missing callback behavior.

## Validation Architecture

### Unit / Service Level

- Verify onboarding dispatch does not use the routine gateway-main session key for interactive onboarding.
- Verify prompt generation or validation treats `NO_REPLY` as invalid for onboarding.
- Verify missing callback transitions onboarding to explicit failure state.

### API Level

- `POST /api/v1/boards/{id}/onboarding/start`
- `POST /api/v1/boards/{id}/onboarding/answer`
- `GET /api/v1/boards/{id}/onboarding`

Need regression coverage that proves the final `GET` no longer remains indefinitely `active` when the gateway session silently ends.

### UI Level

- `frontend/src/components/BoardOnboardingChat.tsx`
- `frontend/src/lib/onboarding.ts`

Need visible stalled/error handling instead of permanent loading copy.

### Live Verification

- Recreate a board onboarding flow against the server after deploy
- Confirm each answer leads to:
  - next question, or
  - final completion, or
  - explicit stalled/error state

Never infinite waiting with no operator-visible reason.
